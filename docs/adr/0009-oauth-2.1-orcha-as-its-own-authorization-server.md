# OAuth 2.1 for `/mcp`: Orcha as its own authorization server, opaque tokens hashed at rest

ADR 0008 shipped `/mcp` PAT-authed and named a single seam — `resolveRole` —
where a richer auth model would later swap in "touching no tool and no resolver."
This is that model. To let a **consumer Claude client** (Claude Desktop, the
claude.ai connector) connect without a hand-pasted token, Orcha becomes its **own
OAuth 2.1 authorization server**: it runs discovery, dynamic client registration,
PKCE authorization, and token issuance itself, reusing the existing login/session
for the consent screen. No managed identity provider sits in the path.

The tokens it issues are **opaque strings hashed at rest** — the same discipline
as a Personal Access Token, never a JWT. Validation is a DB lookup of the SHA-256
hash, so a token can be revoked instantly by flipping a row, and no signing key
is ever minted, stored, or rotated. OAuth gets its **own tables**
(`OAuthClient`, `OAuthAuthorizationCode`, `OAuthAccessToken`, `OAuthRefreshToken`),
distinct from `PersonalAccessToken`; the two families converge only at
`resolveRole`, which dispatches on the token prefix (`orcha_oat_` vs
`orcha_pat_`) and returns the identical `{ role, readOnly, tokenId }` the tools
already depend on. The **PAT path is unchanged** — OAuth is purely additive.

## Why

- **Identity stays first-party.** A managed IdP would put a third party (and, for
  the hosted SaaS, a per-seat bill and a vendor lock-in) between a user and their
  own workspace, and would force every self-hoster to stand up or rent one. Orcha
  already owns login, sessions, and the Role model that carries identity **and**
  tenant in one foreign key (ADR 0006); the authorization server is a thin surface
  over what exists. Consent reuses the logged-in session directly.
- **Don't hand-roll OAuth — but don't import an IdP either.** The Model Context
  Protocol SDK ships the full authorization-server surface (`mcpAuthRouter`:
  metadata, DCR, `/authorize`, `/token`, with PKCE-S256 enforced) behind an
  `OAuthServerProvider` interface. We implement that interface against our token
  stores and render our own consent page; the protocol mechanics are the library's.
- **Opaque + hashed beats JWT here.** A stateless signed token can't be revoked
  without a denylist that reintroduces the very lookup it was meant to avoid — and
  connected-client revocation is a hard requirement, not a nicety. A hashed opaque
  token is revoked by setting `revokedAt`; the next call's DB lookup refuses it.
  It also needs no signing-key lifecycle (generation, storage, rotation, JWKS),
  which is real operational surface a self-hoster would otherwise own. The PAT
  path already validates this way, so there is one storage discipline, not two.
- **Scope is an input; `readOnly` is its materialization.** The vocabulary is just
  `read` and `read write` (write implies read). Consent maps the chosen scope onto
  the existing `readOnly` capability column at mint time, and read-only enforcement
  reuses the shared `writeAs` write-path guard — no new per-tool logic. Because
  `readOnly` is derived from scope and stored alongside it, the two cannot drift.
- **PKCE-S256 + refresh rotation are the 2.1 floor.** Public clients (no secret)
  require PKCE. Short-lived access tokens (~1h) keep a leak small; a rotating
  refresh token (sliding 30 days) keeps a monthly-used client from re-prompting.
  Rotation carries reuse detection: a presented-but-spent refresh token revokes its
  whole **family** (every access and refresh token sharing a `familyId`), the
  OAuth 2.1 theft response.

## Considered options

- **A managed IdP (Auth0, Okta, Cognito, …).** Rejected. Adds a vendor in the
  identity path, a cost per user on the SaaS, and a hard external dependency for
  every self-hoster — to delegate a flow we can run over the login and Role model
  we already own. First-party identity is the whole point of `resolveRole`.
- **JWT / stateless signed access tokens.** Rejected. Buys validation-without-a-DB
  we don't need (the lookup is one indexed hash read) at the cost of a signing-key
  lifecycle and no real revocation — revoking a JWT means a denylist lookup, i.e.
  the DB read again, plus key management on top. Opaque-hashed gives instant
  revocation for free and reuses the PAT storage discipline.
- **Generalize the `PersonalAccessToken` table for both.** Rejected. An OAuth
  access token is client-bound, PKCE-derived, short-lived, refreshable, and lives
  in a rotation family; a PAT is a long-lived, user-minted, standalone secret.
  Forcing both into one table means nullable columns that are required for one
  lifecycle and meaningless for the other, and a row that can't be read without
  knowing which kind it is. Separate tables keep each lifecycle honest; they
  converge where it actually matters — at `resolveRole`.
- **A separate authorization-server process or component.** Rejected for the same
  reason ADR 0008 kept `/mcp` in the backend: consent must read the httpOnly
  session and the Role model, which live here. A second component would re-expose
  both over the network for no gain (ADR 0003 — every component is a cost).
- **A React/SPA consent screen.** Rejected. Only the backend can read the httpOnly
  session (login state + the user's Roles), and the approval must POST same-origin
  for the session cookie to survive (`sameSite` in dev). Consent is therefore
  backend-rendered, styled to match the app; it is the one server-rendered page in
  an otherwise SPA frontend, and that is deliberate.

## Consequences

- **Three token families, one seam.** `orcha_pat_`, `orcha_oat_`, and `orcha_ort_`
  are all opaque and hashed at rest. `resolveRole` dispatches access tokens
  (`oat`/`pat`) by prefix to their store and returns one shape; refresh tokens
  (`ort`) are exchanged only at `/token`. Tools and resolvers see none of this.
- **An authorization-server surface mounts at the app root, behind session
  middleware** (the opposite of `/mcp`, which sits ahead of it): the SDK's
  `/.well-known/*` metadata, `/register` (DCR), `/authorize`, `/token`, plus the
  backend `/oauth/consent` routes. An unauthenticated `/mcp` call returns `401`
  with `WWW-Authenticate: Bearer resource_metadata="…"`, which is how a consumer
  client discovers the AS and bootstraps the whole flow with no manual config.
- **Consumer OAuth requires HTTPS.** `mcpAuthRouter` refuses a non-HTTPS issuer
  except on `localhost`/`127.0.0.1`. A self-hoster must terminate TLS (Traefik
  already does) for browser clients to connect via OAuth; the PAT path has no such
  requirement and remains the zero-TLS option for local/scripted use.
- **Connected-client management falls out of the model.** A "connected app" is a
  `familyId`; listing reads the live grants for a Role, and revoking one calls the
  same `revokeFamily` the reuse-detection path uses — killing its access **and**
  refresh tokens atomically. No separate revocation machinery.
- **One single-instance assumption remains.** The in-flight authorization requests
  awaiting consent are held in an in-process map, so a multi-instance deployment
  would need that moved to shared storage (Redis) before horizontal scale. Issued
  tokens are already in the database and are not affected.
- This realizes the future change ADR 0008 deferred: it lands entirely behind
  `resolveRole` and adds the AS surface, without touching a tool or a resolver.
